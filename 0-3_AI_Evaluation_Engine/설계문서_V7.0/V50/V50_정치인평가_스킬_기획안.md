# Skill 기획안: evaluate-platoon-v50

**기반**: platoon-formation-v3
**목적**: V50 정치인 평가 — 소대(Claude Code 세션) 1개가 20-50명 배치 처리 (분대원 수 탄력적)
**작성일**: 2026-03-07

---

## 스킬 개요

```yaml
description: "V50 소대 단위 정치인 평가 — 소대 1개가 N명씩 배치 처리 (분대원 수 탄력적), 지휘관이 반복 실행"
user-invocable: true
command: /evaluate-platoon-v50
```

**핵심 원칙: 스킬 = 소대 1개**
- 이 스킬은 **소대(Claude Code 세션) 1개의 프로세스**만 정의
- 대규모 처리 = 지휘관이 이 스킬을 **여러 번 실행** (규모 제한 없음)
- 통합 스킬 불필요 — 소대 단위 반복으로 확장

**★ 분대원 수 탄력적 운용 (Claude Code 한도 기반)**

```
플랫폼 동시 실행 한도 분석:
  Claude Code (세션당 background 동시 실행): 최대 10개  ← 핵심 제약
  n8n Cloud Enterprise (동시 workflow):       무제한     ✅
  Claude API Haiku 4.5 Tier3 (RPM):          2,000/분   ✅ (여유)
  Supabase PgBouncer 트랜잭션 모드:           30 연결 풀  ✅ (50+ 클라이언트 처리 가능)

→ 분대원(Haiku Subagent) 최대 10명/분대장 (Claude Code 세션당 한도)
→ 분대장 5명 × 분대원 10명 = 소대당 최대 50명/배치
```

**스킬 시작 시 지휘관이 편제 선택 (2가지 질문):**
```
질문 ①: "분대 수는 몇 개로 편성하시겠습니까? (3-5개, 기본값: 5개)"
질문 ②: "분대원 수는 몇 명으로 편성하시겠습니까? (4-10명, 기본값: 5명)"

  → 소대 총원: M 분대 × N 분대원 = M×N명/배치

  → 예시:
     3분대 × 4명 = 12명/배치  (가볍게)
     5분대 × 5명 = 25명/배치  (기본값)
     5분대 × 7명 = 35명/배치  (빠름)
     5분대 × 10명 = 50명/배치 (최대, Claude Code 한도)
```

```
[사용 도구]

  • Supabase 대시보드 (현황 확인)
```

### 소대장 (Claude Code 세션 1개 = 소대 1개)

```
[하는 일]
  • TeamCreate: platoon-{N} (팀 생성)
  • 지휘관에게 분대 수(M, 3-5개)·분대원 수(N, 4-10명) 확인
  • Supabase에서 pending M×N명 조회 → assigned 처리
  • 분대장 M명 스폰 (Teammate, Sonnet, SendMessage)
  • M×N명 전원 done 확인
  • 배치 완료 후 지휘관에게 3가지 질문
  • pending 소진 또는 해산 승인 시 TeamDelete → 종료

[사용 도구]
  • TeamCreate / TeamDelete (팀 생성/해산)
  • SendMessage (분대장과 통신)
  • Supabase REST API (pending 조회, assigned 처리)
```

### 분대장 × M (Teammate, Sonnet, M=3-5개)

```
[하는 일]
  • 소대장으로부터 N명 배분 수령
  • 분대원 N명 스폰 (Task tool, background=true, Haiku, 최대 10명)
  • N명 완료 확인 → 소대장에게 SendMessage 보고

[사용 도구]
  • Task tool (분대원 Subagent 스폰, background=true)
  • SendMessage (소대장에게 보고)
```

### 분대원 × M×N (Subagent, Haiku, 기본 25명 · 최대 50명)

```
[하는 일]

    → Supabase polling (5초 간격, 완료 대기)

[사용 도구]
```

```
│  "25명 배치 완료: 성공 23명 / 실패 2명                 │
│                                                      │
│  ① 실패 2명 재실행하시겠습니까?                         │
│     → [예] 실패 정치인 재배치 (processing_status 초기화) │
│     → [아니오] 실패 유지                               │
│                                                      │
```

## 스킬 실행 단계 (소대 1개 기준)

> ※ 아래 "단계"는 소대 운영 흐름입니다. n8n 파이프라인의 "Phase 0-5"와 별개입니다.

### 단계 1. 소대장 초기화
```
1. TeamCreate: platoon-{N}
2. Supabase 큐 확인: status='pending' AND eval_week='{week}' 조회
3. pending 목록 확인 → 25명 배치 준비
```

### 단계 1.5. 지휘관에게 보고 (편제 선택 + 개시 확인)
```
소대장이 지휘관에게:
  "pending {P}명 확인.

  편제를 선택하시겠습니까?
  ① 분대 수는 몇 개로 편성하시겠습니까? (3-5개, 기본값: 5개)
  ② 분대원 수는 몇 명으로 편성하시겠습니까? (4-10명, 기본값: 5명)
     → Claude Code background 동시 실행 한도: 최대 10명/분대

  → 소대 총원: M분대 × N명 = M×N명/배치
  M×N명 배치로 개시하시겠습니까?"

→ 승인 시 단계 2 진행 (M, N 값 확정 후 고정)
```

### 단계 2. 배치 실행 (소대장 → 분대장 → 분대원)
```
소대장:
  1. Supabase에서 pending 25명 조회 → assigned 처리
  2. 분대장 5명 스폰 (Teammate, Sonnet, SendMessage) → 5명씩 배분
  3. 각 분대장: 분대원 5명 스폰 (Task tool, Haiku, background=true)
  4. 각 분대원: soldier_v50.py 실행 → n8n webhook POST → polling
```

### 단계 3. n8n 파이프라인 실행 (분대원 1명 = 정치인 1명)

**soldier_v50.py (분대원이 실행):**
```python
# Step 1: n8n webhook 트리거 (시작)
POST {N8N_WEBHOOK_URL}
body: {politician_id, politician_name, week}

# Step 2: Supabase polling (대기)
while status not in ['done', 'failed']:
    time.sleep(5)
    status = supabase.select('processing_status').eq('id', politician_id)

# Step 3: 결과 반환
return {"status": status, "score": score}
```


## 소요시간 정밀 계산

### 정치인 1명 Phase별 구조 (n8n 기준, 시간은 실측 후 확정)

| Phase | 작업 내용 | 비고 |
|-------|---------|------|
| Phase 0 | DB 등록 확인 | Supabase 1회 쿼리 |
| Phase 1 | 3채널 수집 | 3개 HTTP 병렬 |
| Phase 2 | 검증 (Code Node) | 로컬 JS 연산 |
| Phase 2-2 | 재수집 루프 | 평균 1-2 라운드 |
| Phase 3 | 4 AI 평가 | 60 배치 × 4 AI 병렬 |
| Phase 3-1 | 평가 검증 (누락 재평가) | 95% 미달 시만 |
| Phase 4 | 점수 계산 | 로컬 JS 연산 |
| Phase 4.5 | PO 승인 게이트 | CCI 결과 보고 후 PO 승인 대기 (인간 개입 필수) |
| Phase 5 | HTML 보고서 | 템플릿 렌더링 |
| n8n 오버헤드 | 노드 간 전환 | 네트워크 지연 포함 |
| **1명 완료** | | **실측 후 확정** |

### 소대 1개 처리량 (편제별)

| 편제 (M분대 × N명) | 소대 총원/배치 | 배치 수 (P명 처리 시) | 소요 시간 |
|------------------|------------|-------------------|---------|
| 3분대 × 4명 (최소) | 12명 | ceil(P / 12) 배치 | 실측 후 확정 |
| 5분대 × 5명 (기본) | 25명 | ceil(P / 25) 배치 | 실측 후 확정 |
| 5분대 × 7명 (빠름) | 35명 | ceil(P / 35) 배치 | 실측 후 확정 |
| 5분대 × 10명 (최대) | 50명 | ceil(P / 50) 배치 | 실측 후 확정 |

※ P = 처리 대상 정치인 수 (지휘관이 결정)

### 소대 여러 개 병렬 실행 시

```
소대 S개 병렬: ceil(P / (S × M × N)) 배치 × T분/배치

T = 실측값 (n8n 파이프라인 실행 후 측정)
P = 처리 대상 정치인 수 (지휘관이 결정)
S = 병렬 소대 수 (지휘관이 결정)
```

## 상태 관리 (Supabase)

> ⚠️ 참고: 아래 ALTER 문은 소대 편제 운용 시에만 필요합니다. 현재 V50 기본 스키마(`v50_schema.sql`, `v50_schema_supplement.sql`)에는 미포함 상태입니다. 소대 편제 실전 투입 시 별도 마이그레이션으로 적용하세요.

```sql
-- politicians 테이블 추가 컬럼
-- ⚠️ 미적용: processing_status는 v50_schema.sql에 포함됨.
--    assigned_platoon, eval_week는 소대 편제 운용 시에만 필요하며 현재 기본 스키마에 미포함.
ALTER TABLE politicians ADD COLUMN
  processing_status TEXT DEFAULT 'pending',
  -- 값: pending | assigned | phase0_done | phase1_done |
  --     phase2_done | phase22_done | phase3_done |
  --     phase4_done | done | failed
  assigned_platoon TEXT,    -- ⚠️ 미적용 (소대 편제 운용 시 마이그레이션으로 추가)
  eval_week TEXT,           -- '2026-W10' ⚠️ 미적용 (소대 편제 운용 시 마이그레이션으로 추가)
  started_at TIMESTAMPTZ,
  completed_at TIMESTAMPTZ
```

**Supabase 역할 (명확 정의):**
- 상태 저장: 각 정치인의 processing_status 관리
- n8n 상태 수신: n8n이 Phase 완료 시 status 업데이트
- Storage: HTML 보고서 저장
- REST API: n8n에서 직접 호출
- Realtime: 대시보드 실시간 업데이트 (WebSocket)
