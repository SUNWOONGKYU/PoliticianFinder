# V50 스크립트 역할 구분 — 누가 무엇을 실행하는가

**작성일**: 2026-03-08
**핵심 원칙**: 실행 주체별로 파일을 명확히 구분

> ⚠️ **절대 오해 금지**
> - **n8n은 수집/평가 안 한다** → 프로세스 관리자(오케스트레이터)만
> - **Python은 수집/평가 안 한다** → 트리거 + polling + 점수계산만
> - **수집 = Gemini/Grok/Naver API가** Instructions를 받아 직접 수행
> - **평가 = 4개 AI API가** Instructions를 받아 각자 독립적으로 수행 (합의 없음!)

## 1. 실행 주체 3가지

```
① Python    — 직접 실행 가능한 .py 파일 (터미널 또는 Claude Code에서 실행)
② n8n       — n8n 워크플로우 안에서 실행 (JavaScript Code 노드 + HTTP Request 노드)
③ AI (API)  — n8n이 HTTP Request로 호출하는 외부 AI 서비스
```

### ② n8n이 관리(오케스트레이션)하는 것 (`V50/n8n/`)

> ⚠️ **n8n은 직접 수집하거나 평가하지 않는다. n8n = 파이프라인 관리자(오케스트레이터)**
> - 수집은 **Gemini/Grok/Naver API가** Instructions를 받아 직접 수행
> - 평가는 **4개 AI API가** Instructions를 받아 각자 독립적으로 수행 (합의 없음!)
> - n8n의 역할: ① Instructions DB 조회 ② HTTP body 임베드 ③ 외부 API 전송 ④ 결과 DB 저장 ⑤ 상태 관리

| 파일 | 내용 | 실행 언어 |
|------|------|---------|
| `v50_single_politician_workflow.json` | Phase 0~5 전체 파이프라인 관리 | **JavaScript** (Code 노드) + HTTP Request |
| `v50_api_config.json` | API 엔드포인트·모델·인증 설정 | 설정값 (n8n 참조용) |

**n8n이 실제로 하는 일 (프로세스 관리만):**

```
[Webhook Trigger]   ← soldier_v50.py가 POST로 호출
     ↓
[Phase 0] Code 노드 (JS): Supabase REST → 정치인 등록 확인
     ↓
[Phase 1] collect_instructions 조회 → HTTP body 임베드 → HTTP Request × 3 병렬 전송
            ↳ Gemini API가 직접 데이터 수집하여 반환 (40%, 48개/카테고리)
            ↳ Grok API가 직접 X/트위터 수집하여 반환 (10%, 12개/카테고리)
            ↳ Naver API가 직접 뉴스 수집하여 반환   (50%, 60개/카테고리)
     ↓
[Phase 2] Code 노드 (JS): 중복·기간·비율 검증 (이건 n8n JS가 직접 처리)
     ↓
[Phase 2-2] Execute Workflow (최대 4회 재수집 요청 — API에게)
     ↓
[Phase 3] eval_instructions 조회 → HTTP body 임베드 → SplitInBatches(25) + HTTP Request × 4 병렬
            ↳ Claude Haiku가 독립적으로 평가하여 반환  (n8n이 평가하는 게 아님!)
            ↳ Gemini Flash-Lite가 독립적으로 평가하여 반환
            ↳ ChatGPT gpt-4o-mini가 독립적으로 평가하여 반환
            ↳ Grok grok-3-mini가 독립적으로 평가하여 반환
            ※ 4개 AI는 합의하지 않는다. 각자 독립 판정 후 Phase 4에서 평균!
     ↓
[Phase 4] Code 노드 (JS): 4개 AI 독립 평가 결과 평균 → Bayesian Prior 점수 계산
            category_score = (6.0 + avg_score × 0.5) × 10  ← "6.0"이 Bayesian Prior
     ↓
[Phase 4.5] PO 승인 게이트 (CCI 결과 보고 → PO 승인 대기, 인간 개입 필수)
     ↓
[Phase 5] Code 노드 (JS): HTML 보고서 → Supabase Storage
     ↓
[Supabase] processing_status = 'done'
```

### ③ AI API — 실제 수집/평가 담당 (n8n이 Instructions+요청을 전달, API가 직접 수행)

> ⚠️ **이것이 진짜 일꾼들이다.** n8n은 이들에게 지시(Instructions + 데이터)를 전달하는 파이프일 뿐.

**수집 API (Phase 1에서 실제 수집 수행):**

| 서비스 | 실제 역할 | 수집 목표 | 모델/방식 |
|--------|----------|---------|----------|
| **Gemini API** | **직접 뉴스/공개자료 수집** | 48개/카테고리 (40%) | gemini-2.0-flash-lite |
| **Grok Live Search** | **직접 X/트위터 실시간 수집** | 12개/카테고리 (10%) | grok-3-mini (web_search_preview) |
| **Naver NCP API** | **직접 뉴스 검색** | 60개/카테고리 (50%) | REST API |

**평가 API (Phase 3에서 각자 독립 평가 수행 — 합의 없음!):**

| 서비스 | 실제 역할 | 배치 | 모델 |
|--------|----------|------|------|
| **Claude Haiku API** | **독립 평가** (+4~-4 판정) | 25개 | claude-haiku-4-5-20251001 |
| **Gemini API** | **독립 평가** (+4~-4 판정) | 25개 | gemini-2.0-flash-lite |
| **ChatGPT API** | **독립 평가** (+4~-4 판정) | 25개 | gpt-4o-mini |
| **Grok API** | **독립 평가** (+4~-4 판정) | 25개 | grok-3-mini |

**4개 AI 평가 방식:**
```
각 AI가 독립적으로 rating을 판정 → DB 저장
Phase 4에서 4개 AI 결과를 평균 계산
= 합의(consensus)나 투표(voting)가 아님!
= 4개 AI 독립 관측값의 수치 평균
```

> AI API는 별도 "스크립트"가 없다. **n8n HTTP Request 노드가 Instructions를 포함한 요청을 전달**하면, **AI API가 직접 수집/평가를 수행하여 결과를 반환**한다.
>
> **[Supabase REST]** 상태 저장/조회 — 전 Phase에서 n8n이 사용

## 3. Instructions — 스크립트인가, 참조 콘텐츠인가?

### 결론: Instructions는 **콘텐츠(프롬프트 원고)**이다. 스크립트가 아니다.

```
instructions/
  ├── 1_politicians/{이름}.md    ← 정치인 기본 정보 (Phase 0에서 DB 등록 시 참조)
  ├── 2_collect_v50/             ← 수집 프롬프트 원고
  │   ├── cat01_expertise.md
  │   ├── cat02_leadership.md
  │   └── ... (10개)
  └── 3_evaluate_v50/            ← 평가 기준 원고
      ├── cat01_expertise.md
      ├── cat02_leadership.md
      └── ... (10개)
```

### Instructions가 실제로 사용되는 방식

```
[2_collect_v50/catNN_*.md]
  → n8n Phase 1이 Supabase에서 해당 카테고리 수집 지침을 조회
  → HTTP Request 프롬프트 body에 내용 임베드
  → Gemini/Grok API로 전송 (수집 프롬프트)

[3_evaluate_v50/catNN_*.md]
  → n8n Phase 3이 Supabase에서 해당 카테고리 평가 기준을 조회
  → HTTP Request 프롬프트 body에 내용 임베드
  → Claude/Gemini/ChatGPT/Grok API로 전송 (평가 프롬프트)

[1_politicians/{이름}.md]
  → Phase 0에서 사람이 직접 참조하거나
  → n8n Phase 0이 Supabase에서 정치인 정보 확인 시 참조
```

### 수집 Instructions vs 평가 Instructions — 반드시 분리!

| 구분 | 파일 위치 | 용도 | 절대 금지 |
|------|---------|------|---------|
| **수집 지침** | `2_collect_v50/catNN_*.md` | "어떤 데이터를 찾을지" | 평가 기준 포함 금지 |
| **평가 기준** | `3_evaluate_v50/catNN_*.md` | "어떻게 등급을 매길지" | 수집 방법 포함 금지 |

> 수집할 때 평가 기준을 보면 → **수집 편향** 발생
> 평가할 때 수집 방법을 보면 → **불필요한 컨텍스트** 낭비

## 5. 요약 (핵심)

```
누가 무엇을 하는가?

Python(soldier_v50.py)  → n8n에 시작 신호 보내고, 완료될 때까지 polling. 점수 계산(비상용).
n8n                     → 프로세스 관리자. Instructions 조회 → API에 전달 → 결과 DB 저장.
Gemini/Grok/Naver API   → Instructions 받아 실제 데이터 수집. n8n이 수집하는 게 아님!
Claude/Gemini/ChatGPT/Grok → Instructions 받아 각자 독립 평가. 합의 없음, 평균만 함.
Instructions            → 스크립트 아님. 수집/평가 API에게 주는 프롬프트 원고.
```

| 주체 | 하는 일 | 하지 않는 일 |
|------|---------|------------|
| **Python(soldier)** | webhook POST, DB polling | 수집, 평가 |
| **n8n** | 프로세스 관리, Instructions 전달 | 수집, 평가 |
| **수집 API(3개)** | 실제 데이터 수집 | 평가 |
| **평가 AI(4개)** | 각자 독립 평가 | 수집, 합의 |
| **Python(비상용 12개)** | n8n 대체 수동 실행 | 정상 운용 시 사용 안 함 |
