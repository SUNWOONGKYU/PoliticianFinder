## 1. 시스템 발명 요소 정리

특허 출원 전에 이 시스템이 무엇을 발명했는지를 명확히 정의해야 한다.
아래는 V50 시스템의 7개 핵심 기술 요소를 정리한 것이다.

### 발명 요소 A: 다중 AI 독립 평가 + 수치 평균 집계

```
구성:
  4개 AI (Claude, Gemini, ChatGPT, Grok)가
  동일한 데이터 집합에 대해 각자 독립적으로 평가를 수행.
  평가 중 AI 간 합의(consensus) 또는 투표(voting) 없음.
  각 AI는 동일한 평가 기준(Instructions)을 입력받아
  +4 ~ -4 등급(총 8단계, 0 없음)의 정성적 점수를 부여.
  Phase 4에서 4개 AI의 점수를 단순 수치 평균하여 최종값 산출.

기술적 특이점:
  - 합의형(ensemble consensus)이 아닌 독립 관측 + 사후 평균
  - 평가 AI와 집계 로직이 완전히 분리된 아키텍처
  - 0점을 제외한 비대칭 8단계 등급 체계 (+4,+3,+2,+1,-1,-2,-3,-4)
```

### 발명 요소 B: 수집/평가 Instructions 물리적 분리 저장

```
구성:
  수집 지침 (collect_instructions): Supabase 테이블 독립 저장
  평가 기준 (eval_instructions):   Supabase 테이블 독립 저장
  두 지침은 서로 다른 Phase에서만 참조되며, 교차 참조 불가.

  Phase 1 (수집): 오직 collect_instructions만 로드 → AI에 전달
  Phase 3 (평가): 오직 eval_instructions만 로드 → AI에 전달

기술적 특이점:
  - 수집 편향(collection bias) 구조적 차단
    (평가 기준을 모르는 상태에서 수집 → 수집자가 유리한 데이터만
     선택하는 편향 방지)
  - DB 물리적 분리로 단순 프롬프트 분리가 아닌 스키마 레벨 격리
  - Audit 가능: 각 지침의 버전 이력이 DB에 별도 보관
```

### 발명 요소 C: Bayesian Prior 기반 점수 공식

```
공식:
  category_score = (PRIOR + avg_score × COEFFICIENT) × 10
  where PRIOR = 6.0, COEFFICIENT = 0.5
  avg_score = 4개 AI × 각 항목 rating의 평균 (단위: 점, 범위 -8~+8)

  final_score = round(min(Σ(10 categories), 1000))
  범위: 200 ~ 1000점

Bayesian Prior 역할:
  데이터가 없거나 부족한 정치인의 경우,
  avg_score → 0으로 수렴하여 category_score = 60점
  (PRIOR × 10 = 6.0 × 10 = 60점)이 되는 사전확률 역할.
  데이터 부족 카테고리에 대해 0점이 아닌 중립값 자동 부여.

기술적 특이점:
  - 전통적 정치인 평가 지수에 없는 Bayesian Prior 개념 적용
  - 평가 불가 항목(데이터 <25개)을 leverage score 0으로 처리,
    공식에서 avg_score=0으로 수렴시켜 60점 산출
    (하드코딩 아님, 수식 유도 결과)
  - 점수 범위 200~1000 (최저 60점×10 카테고리 = 600이 아니라
    각 카테고리 20점 하한선 설계 포함)
```

### 발명 요소 D: Sentiment 비율 강제 제어

```
구성:
  각 카테고리별 수집 데이터를 3가지 감성으로 분류:
    - negative (부정): 비리, 실패, 논란 관련
    - positive (긍정): 성과, 공약 이행 관련
    - free    (자유): 중립/혼합

  비율 강제 규칙:
    Phase 1: 부정 20% 이상 보장 (수집 단계)
    Phase 2: 비율 검증 → 미달 시 Phase 2-2 재수집 트리거
    Phase 2-2: 최대 4회 반복, 미달 카테고리만 재수집

기술적 특이점:
  - 긍정 편향 구조적 방지 (찬사 데이터만 쌓이는 현상 차단)
  - 카테고리별 독립 비율 관리 (전체 비율이 아닌 카테고리 단위)
  - Give-Up 기준 수식화: 4회 재수집 후 25개 미만 → leverage 0
```

### 발명 요소 E: 소대 편제 계층형 에이전트 병렬 처리

```
구조 (3계층):
  소대장 (Claude Code 세션 1개)
    → 분대장 × M (Teammate, M=3~5개)
    → 분대원 × M×N (Subagent, N=4~10명/분대)

  정치인 1명 = 분대원 1명 담당
  분대원 실행: soldier_v50.py → n8n webhook POST → Supabase polling

처리 능력:
  소대 1개 = M×N명 동시 처리 (최대 50명/배치)
  소대 K개 병렬 = K×M×N명 동시 처리
  규모 제한 없음 (지휘관이 소대 추가 실행으로 확장)

기술적 특이점:
  - 군사 편제 개념을 AI 에이전트 오케스트레이션에 적용
  - Supabase processing_status를 분산 락(distributed lock)으로 활용
    (assigned 처리로 중복 배정 방지)
  - 3가지 질문 루프: 배치 완료 후 재실행/추가/해산 의사결정 자동화
```

### 발명 요소 F: n8n 오케스트레이터 + 외부 AI API 이중 구조

```
구성:
  n8n (오케스트레이터): 파이프라인 관리만 담당
    - Instructions 조회 (Supabase)
    - HTTP body 임베드
    - 외부 AI API에 전달
    - 결과 DB 저장
    - 상태(processing_status) 관리

  외부 AI API (실제 실행):
    - 수집: Gemini API / Naver NCP / Grok Live Search
    - 평가: Claude Haiku / Gemini Flash-Lite / ChatGPT gpt-4o-mini / grok-3-mini

기술적 특이점:
  - n8n이 직접 수집/평가하지 않음 → 오케스트레이터/실행자 역할 분리
  - 수집 채널과 평가 AI가 완전히 다른 조합 가능 (조합 가변성)
  - SplitInBatches(25) + HTTP × 4 병렬로 평가 처리율 극대화
```

### 발명 요소 G: OFFICIAL/PUBLIC 기간 분리 수집

```
구성:
  OFFICIAL (공식 자료): 정부/의회 공식 기록 → 최근 4년 이내
  PUBLIC   (공개 자료): 뉴스/SNS/시민 반응  → 최근 2년 이내

  두 카테고리는 서로 다른 기간 기준 적용.
  기간 초과 데이터는 Phase 2 검증에서 자동 제거.

기술적 특이점:
  - 동일 정치인에 대해 공식성(officiality)에 따른 이중 기간 필터
  - 오래된 공식 기록의 높은 신뢰성과 최근 여론의 현재성을 동시 반영
  - 카테고리별 OFFICIAL/PUBLIC 비율 목표치 독립 설정 가능
```

## 2. 특허 가능성 분석

### 2.1 분석 개요

소프트웨어/AI 특허는 국가별로 기준이 크게 다르다. 특히 "추상적 아이디어"에 해당하는지 여부가 핵심 관문이다. 아래에서는 한국(KIPO), 미국(USPTO), EU(EPO) 3개 관할의 기준으로 7개 발명 요소를 분석한다.

---

### 2.2 한국 특허청(KIPO) 기준 분석

**관련 규정**: 특허법 제2조 제1호, 컴퓨터 관련 발명 심사기준(2022 개정)

한국은 소프트웨어가 하드웨어와 결합하여 구체적인 기술적 수단으로 구현되면 특허 대상으로 인정한다. "소프트웨어에 의한 정보처리가 하드웨어를 이용해 구체적으로 실현되는 것"이 요건이다.

| 발명 요소 | 신규성 | 진보성 | 산업상 이용가능성 | 종합 판단 |
|-----------|--------|--------|-----------------|----------|
| **A: 다중 AI 독립 평가 + 수치 평균** | 높음 — 정치인 대상 다중 LLM 독립 평가 + 평균 집계 선행기술 미확인 | 높음 — AI 합의형 앙상블과 구별되는 독립 관측 방식 | 명확 — 정치인 평가 서비스 산업 직접 적용 | **출원 강력 추천** |
| **B: 수집/평가 Instructions 물리적 분리** | 중간 — 프롬프트 분리 개념 자체는 일반적, 그러나 DB 레벨 물리 격리 + 수집편향 방지 목적의 구조화는 신규성 주장 가능 | 중간-높음 — 수집편향 방지를 위한 구조적 아키텍처 설계로 진보성 인정 가능성 | 명확 | **출원 추천 (청구항 설계 중요)** |
| **C: Bayesian Prior 점수 공식** | 중간 — Bayesian Prior 자체는 공지, 그러나 정치인 평가 + 데이터 부족 처리 특정 공식은 신규성 주장 가능 | 중간 — 수학적 공식의 진보성은 "특정 기술 문제 해결" 입증 필요 | 명확 | **출원 가능 (구체적 적용 맥락 강조 필요)** |
| **D: Sentiment 비율 강제 제어** | 높음 — AI 수집 데이터의 감성 비율을 카테고리별로 강제 제어하는 방법 선행기술 미확인 | 높음 — 수집 편향 방지라는 기술적 과제 해결 | 명확 | **출원 강력 추천** |
| **E: 소대 편제 계층형 에이전트 병렬 처리** | 중간-높음 — 멀티에이전트 계층 구조 자체는 선행기술 존재, 그러나 군사 편제 + Supabase 분산락 + 3질문 루프 특정 조합은 신규성 주장 가능 | 중간 | 명확 | **종속항으로 보호 추천** |
| **F: n8n 오케스트레이터 + 외부 AI API 이중 구조** | 낮음-중간 — n8n 오케스트레이션 자체는 공지기술, 수집/평가 역할 분리 + 특정 AI 조합은 차별성 주장 여지 | 낮음 — 구성의 조합이 자명할 수 있음 | 명확 | **독립항보다 종속항으로 포함** |
| **G: OFFICIAL/PUBLIC 기간 분리 수집** | 중간 — 이중 기간 필터 자체는 개념적, 구체적 구현(4년/2년 기준, DB 필드 설계) 포함 시 신규성 주장 가능 | 중간 | 명확 | **출원 가능 (구체적 구현 강조 필요)** |

**한국 기준 종합 평가**: 발명 요소 A, D가 가장 강력. B, C, E, G는 청구항 설계에 따라 보호 가능.

### 2.3 미국 USPTO 기준 분석

**주요 판례 장벽**: Alice Corp. v. CLS Bank (2014) — 추상적 아이디어에 불과한 소프트웨어 특허 불인정. Mayo Collaborative v. Prometheus (2012) — 자연법칙 적용만으로는 특허 불가.

**Alice 2단계 테스트**:
- Step 1: 청구항이 특허 대상(기계/제조물/방법/조성물)에 해당하는가?
- Step 2A: 사법부 예외(추상적 아이디어, 자연법칙, 자연현상)에 해당하는가?
  - Step 2B: 그럼에도 청구항이 "훨씬 더 많은(significantly more)" 것을 포함하는가?

| 발명 요소 | Alice Step 2A | Alice Step 2B ("significantly more") | USPTO 전망 |
|-----------|--------------|--------------------------------------|-----------|
| **A: 다중 AI 독립 평가 + 수치 평균** | 추상적 아이디어에 해당할 위험 ("여러 전문가 의견 평균"의 소프트웨어 구현) | 특정 AI 모델 조합 + 비대칭 등급 체계 + 수집-평가 분리 아키텍처가 추상적 아이디어를 넘어서는 구체적 기술 구현임을 주장 가능 | **중간-긍정적** (청구항 설계 매우 중요) |
| **B: 수집/평가 Instructions 물리적 분리** | DB 스키마 설계로 구체적 기술 수단 주장 가능 | DB 레벨 물리 격리로 수집편향 방지라는 기술적 문제 해결 — significantly more 인정 가능성 있음 | **긍정적** |
| **C: Bayesian Prior 점수 공식** | 수학적 관계식 자체는 추상적 아이디어 | 정치인 데이터 부족 상황의 특정 문제(leverage score 0 처리)와 결합된 구현이 필요 | **어려움** (방어적 접근 필요) |
| **D: Sentiment 비율 강제 제어** | 데이터 필터링/분류 방법 — 추상적 아이디어 위험 | 카테고리별 독립 비율 관리 + 자동 재수집 트리거라는 컴퓨터 시스템의 구체적 개선 주장 가능 | **중간-긍정적** |
| **E: 소대 편제 에이전트 시스템** | 계층형 작업 분배 — 추상적 아이디어 위험 | Supabase 분산 락 + 특정 Claude Code API 구현이 구체적 기술 수단 | **중간** |
| **F: 이중 구조 오케스트레이터** | 범용 오케스트레이션 도구 사용 — 위험 | 수집/평가 분리 + 특정 HTTP 배치 처리 구조로 보강 필요 | **어려움** |
| **G: OFFICIAL/PUBLIC 기간 분리** | 날짜 필터링 — 추상적 아이디어 위험 | 공식성 분류 기준 + 자동 검증 + DB 필드 설계의 결합 | **중간** |

**USPTO 전략 결론**:
- 청구항을 순수 "방법"이 아닌 "시스템"(특정 하드웨어 구성 포함) 또는 "컴퓨터 프로그램 제품"으로 작성할 것.
- Alice 장벽 우회를 위해 "컴퓨터 관련 발명이 컴퓨터 기능 자체를 개선한다"는 논리 구성이 효과적.
- 발명 요소 A(다중 AI 독립 평가)와 B(Instructions 물리적 분리)의 조합이 가장 강력한 청구 기반.

### 2.4 EU EPO 기준 분석

**관련 조항**: EPC Article 52(2)(c) — 정신적 행위, 수학적 방법, 비즈니스 방법 등은 특허 불가. 그러나 "기술적 특성(technical character)"이 있으면 예외.

**EPO 핵심 기준**: 청구항이 "기술적 과제"를 해결하는 "기술적 수단"을 사용하는가?

| 발명 요소 | 기술적 과제 | 기술적 수단 | EPO 전망 |
|-----------|-----------|-----------|---------|
| **A: 다중 AI 독립 평가** | 단일 AI 편향 문제 해결 | 복수 LLM API 병렬 호출 + 결과 집계 시스템 | **긍정적** — 기술적 수단 명확 |
| **B: Instructions 물리적 분리** | 수집편향 방지라는 데이터 품질 기술 문제 | DB 스키마 설계 + 접근 제어 기술 | **긍정적** |
| **C: Bayesian Prior 공식** | 데이터 부족 시 오류 방지 | 특정 수학적 공식의 소프트웨어 구현 — EPO는 수학적 방법도 기술적 효과 있으면 인정 가능 | **중간** |
| **D: Sentiment 비율 제어** | AI 수집 데이터 품질 보증 | 자동 비율 검증 + 재수집 트리거 시스템 | **긍정적** |
| **E: 소대 편제 에이전트** | 대규모 병렬 처리 효율성 | 계층형 멀티에이전트 시스템 + 분산 락 구현 | **중간-긍정적** |
| **F: n8n 이중 구조** | 수집/평가 확장성 | n8n 워크플로우 + 외부 API 통합 아키텍처 | **중간** |
| **G: OFFICIAL/PUBLIC 기간 분리** | 데이터 시의성 보증 | 이중 기간 필터 + 자동 검증 시스템 | **중간-긍정적** |

**EPO 전략 결론**:
- EPO는 USPTO보다 소프트웨어 특허에 우호적.
- "기술적 효과"를 명확히 설명하면 수학적 공식(C)도 보호 가능.
- 한국 출원 기반으로 PCT 국제출원 후 EPO 진입이 효율적.

## 3. 핵심 청구항 설계

### 3.1 독립항 설계 원칙

독립항은 발명의 핵심을 최광의(最廣義)로 청구해야 한다. 지나치게 좁으면 우회가 쉽고, 지나치게 넓으면 거절된다. 아래에 방법 특허, 시스템 특허, 컴퓨터 프로그램 특허 각각의 독립항 초안을 제시한다.

### 3.5 종속항 설계

독립항 1을 기준으로 주요 종속항 초안:

**[종속항 6]** 독립항 1에 있어서, 상기 등급 체계는 양의 정수값(+1 내지 +4) 및 음의 정수값(-1 내지 -4)을 포함하되 0을 포함하지 않는 8단계 비대칭 등급 체계인 방법.

**[종속항 7]** 독립항 1에 있어서, 상기 점수 산출 공식은 `(PRIOR + avg_score × COEFFICIENT) × SCALE`의 형태를 가지며, PRIOR는 사전 정의된 사전확률 기준값으로서 데이터 부족 시 중립 점수를 보장하는, 방법.

**[종속항 8]** 독립항 1에 있어서, 상기 데이터 수집 단계는 수집된 데이터를 부정(negative), 긍정(positive), 자유(free) 감성으로 분류하고, 각 평가 카테고리별로 부정 감성 데이터의 비율이 사전 정의된 최소 비율 이상이 되도록 강제하는 비율 제어 단계를 포함하는, 방법.

**[종속항 9]** 종속항 8에 있어서, 상기 비율 제어 단계는 최소 비율 미충족 시 해당 카테고리에 대한 추가 수집을 요청하는 재수집 트리거 단계를 포함하고, 사전 정의된 최대 재수집 횟수를 초과한 이후에도 미충족 시 해당 카테고리의 avg_score를 0으로 처리하는, 방법.

**[종속항 10]** 독립항 1에 있어서, 상기 복수의 데이터 소스는 공식 자료(OFFICIAL) 및 공개 자료(PUBLIC)를 포함하며, 공식 자료에 대한 수집 허용 기간과 공개 자료에 대한 수집 허용 기간이 서로 상이한, 방법.

**[종속항 11]** 독립항 3에 있어서, 상기 독립 평가 요청부는 복수의 데이터 항목을 사전 정의된 배치 크기로 분할하여 상기 제1 내지 제N 인공지능 언어 모델에 병렬로 전송하는 배치 처리 기능을 포함하는, 시스템.

**[종속항 12]** 독립항 4에 있어서, 상기 최상위 오케스트레이터 에이전트는 일괄 배치 완료 후 재실행 여부, 추가 배치 여부, 종료 여부에 관한 의사결정 인터페이스를 사용자에게 제공하는, 시스템.

| 출원 순서 | 내용 | 핵심 청구 |
|---------|------|---------|
| 1순위 | 다중 AI 독립 평가 + Instructions 물리 분리 (A+B 결합) | 독립항 1, 3, 5 |
| 2순위 | Sentiment 비율 강제 제어 방법 (D) | 독립항 2 + 종속항 8, 9 |
| 3순위 | 계층형 에이전트 병렬 평가 시스템 (E) | 독립항 4 + 종속항 12 |
| 4순위 | OFFICIAL/PUBLIC 이중 기간 분리 수집 (G) | 독립항 1의 종속항 10 |

**출원 형태**: 출원 건수를 최소화하면서 보호 범위를 최대화하기 위해 1순위와 2순위를 1건으로 묶어 출원하는 방법을 변리사와 검토한다.

**우선심사 신청 고려**: 특허청 우선심사 제도를 활용하면 심사를 앞당길 수 있다 (대기업 대비 중소기업 우대). 통상 6~12개월 단축 효과.

---

### 4.3 PCT 국제출원 타임라인

```
T+0:    한국 국내 출원 (우선일 확보)
T+6개월: 국내 출원 심사 첫 의견통지 예상 (조기 출원 시 빠를 수 있음)
T+12개월: PCT 국제출원 신청 기한 (한국 출원일로부터 정확히 12개월)
           ↳ 이 기한 초과 시 PCT 우선권 소멸 — 절대 기한!
T+18개월: 국제 공개 (자동)
T+30개월: 각 국가별 진입 기한 (PCT 우선일 기산)
           ↳ 진입 국가: 미국(USPTO), EU(EPO), 일본, 중국 등

한국 특허 등록: 출원일로부터 평균 18~24개월
미국 특허 등록: 출원일로부터 평균 24~36개월
```

**PCT 출원 관할**: 한국 특허청(KIPO)에서 PCT 국제출원 수리기관으로 직접 접수 가능.

### 4.4 분할출원 전략

분할출원은 원출원에서 일부 청구항을 별도 출원으로 분리하는 것이다. 다음 상황에서 유용하다:

- **원출원 거절 대응**: 독립항 1이 거절 결정을 받았을 때, 종속항 6~12을 별도 독립항으로 분할 출원.
- **청구항 보호 범위 확장**: 초기 출원에 포함하지 못한 추가 구현 방식을 원출원 기간 중 분할.
- **사업 확장 시**: 소대 편제 시스템(E)을 다른 도메인(의사/변호사 평가 등)에 적용할 때 별도 출원.

**분할 가능 기한**: 원출원 특허 등록 결정 전까지(한국 기준).

## 6. 영업비밀 vs 특허 판단

### 6.1 판단 기준 비교

| 기준 | 특허 | 영업비밀 |
|------|------|---------|
| 보호 방법 | 공개 대가로 독점권 부여 | 비공개 유지로 무기한 보호 |
| 보호 기간 | 20년 (한국/미국/EU) | 비밀 유지하는 한 무기한 |
| 침해 입증 | 용이 (등록 내용으로 비교) | 어려움 (독자 개발 항변 인정) |
| 역엔지니어링 | 특허 청구범위 해석으로 대응 | 역엔지니어링에 무방비 |
| 출원 비용 | 높음 (위 5절 참조) | 거의 없음 (관리 비용만) |
| 우회 발명 | 경쟁자가 청구범위 우회 가능 | 독립 개발 시 보호 불가 |

---

### 6.2 발명 요소별 특허/영업비밀 판단

**특허 출원 추천 요소**:

| 요소 | 추천 방식 | 이유 |
|------|---------|------|
| **A: 다중 AI 독립 평가 + 수치 평균** | 특허 | 서비스 론칭 후 경쟁자가 역엔지니어링하기 쉬움. 특허 등록 후 명시적 배타권 행사가 훨씬 효과적. |
| **B: Instructions 물리적 분리** | 특허 | DB 스키마 구조는 서비스 API를 분석하면 유추 가능. 선제 특허가 우위. |
| **D: Sentiment 비율 강제 제어** | 특허 | 경쟁 서비스가 동일 구조를 채택할 가능성 높음. 특허로 차별화. |

**영업비밀 유지 추천 요소**:

| 요소 | 추천 방식 | 이유 |
|------|---------|------|
| **C: Bayesian Prior 구체적 파라미터 값** | 영업비밀 | PRIOR=6.0, COEFFICIENT=0.5의 구체적 수치는 특허 청구 시 공개됨. 수치는 비밀 유지, 공식 구조만 특허 출원. |
| **F: n8n 워크플로우 상세 설계** | 영업비밀 | 특허 가능성 낮고, n8n 설정 파일 자체는 서비스 운영 노하우. 비공개 유지. |
| **Sentiment 분류 세부 규칙** | 영업비밀 | 각 카테고리별 세부 비율(negative 20% 이상 등)의 구체적 수치. 공식 구조는 특허, 수치는 비밀 유지. |
| **수집 Instructions 세부 내용** | 영업비밀 | 각 카테고리별 수집 프롬프트 원고. 구조는 특허 출원, 내용은 영업비밀. |
| **평가 Instructions 세부 내용** | 영업비밀 | 각 카테고리별 등급 판정 기준 원고. 구조는 특허 출원, 내용은 영업비밀. |

**혼합 전략 추천 요소**:

| 요소 | 추천 방식 | 이유 |
|------|---------|------|
| **E: 소대 편제 시스템** | 특허 (구조) + 영업비밀 (파라미터) | 계층형 에이전트 구조는 특허 출원. 분대원 수, 배치 크기 등 최적 파라미터는 영업비밀. |
| **G: OFFICIAL/PUBLIC 기간 기준** | 특허 (이중 기간 구조) + 영업비밀 (구체적 기간값) | 이중 기간 필터 구조는 특허, 4년/2년이라는 구체적 수치는 영업비밀. |

### 6.3 영업비밀 보호를 위한 실무 조치

영업비밀로 유지하기로 결정한 요소들에 대해서는 다음 조치가 필요하다:

1. **비밀유지 계약(NDA)**: 개발자, 협력업체, 투자자 등 모든 접근자에게 NDA 체결.
2. **접근 권한 관리**: Supabase Row Level Security(RLS) + API Key 접근 제어.
3. **비밀 취급 절차 문서화**: 내부 "영업비밀 취급 규정" 수립 — 법적 보호 요건.
4. **형사 고발 요건 구비**: 한국 부정경쟁방지법 제18조 대응 자료 사전 준비.

- [ ] **개발일 증명 자료 확보**: git 커밋 이력, 설계문서 작성 날짜, Supabase 데이터 생성 로그 등 — 선사용 항변 및 진보성 입증에 활용.

- [ ] **AI/소프트웨어 특허 전문 변리사 선임**: 면담 후 계약. 최소 3개 변리사 사무소 비교 검토.
- [ ] **선행기술 조사 의뢰**: 8절의 키워드로 KIPRIS + USPTO 조사 의뢰 (또는 직접 수행).
- [ ] **기술 설명서 초안 작성**: 본 문서의 1절 "발명 요소 정리"를 기반으로 변리사와 협의용 기술 설명서 작성.
- [ ] **청구항 초안 협의**: 본 문서의 3절 독립항 초안을 변리사와 검토.

**중기 (1~3개월)**:

- [ ] **국내 1순위 출원 완료** (A+B+D 결합, 방법+시스템+CRM 3종 청구).
- [ ] **PCT 출원 계획 수립**: 국내 출원일 기준 12개월 이내 PCT 출원 일정 확정.
- [ ] **진입 국가 선정**: 서비스 진출 예정 국가(미국, 일본 등) 기준으로 PCT 이후 진입 국가 확정.

## 결론 및 권고사항

### 핵심 권고

1. **즉시 변리사 선임**: AI/소프트웨어 특허 전문가와 협의 시작. 공개 이력 확인이 최우선.

2. **가장 강력한 특허는 A+B의 결합**: "다중 AI 독립 평가"와 "Instructions 물리적 분리"의 결합이 핵심 청구의 중심이 되어야 한다. 이 두 요소가 결합되면 경쟁자가 하나만 우회해도 나머지 요소로 침해가 성립할 수 있어 방어력이 높다.

3. **특허는 구조, 영업비밀은 수치**: 공식의 구조(사전확률 포함 점수 공식)는 특허로 보호하고, PRIOR=6.0 같은 구체적 파라미터는 영업비밀로 유지한다. 이 전략이 공개 의무(특허)와 경쟁 우위 유지(영업비밀)를 동시에 달성하는 최선이다.

4. **PCT는 국내 등록 후 결정**: 비용 부담이 크므로 국내 출원 심사 결과를 보고 PCT 진행 여부를 결정해도 늦지 않다. 단, 12개월 기한은 절대로 놓치지 말 것.
