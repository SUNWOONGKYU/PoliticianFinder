## API Rate Limit 병목 분석

### Gemini CLI (최대 병목)

```
문제: 55개 분대원이 동시에 Gemini CLI 실행 → 동일 Google 계정 quota 공유

현상:
  - 1개 CLI: ~27초/라운드
  - 55개 동시: 대부분 quota 초과로 fallback 발생

해결책 (3가지):
  A. 멀티 Gemini API 키 로테이션 (키 배포)
     - 분대원 번호 % 5 → 5개 API 키 중 하나 선택
     - Tier 1 기준: 300 RPM × 5키 = 1,500 RPM 확보

  B. collect_gemini_subprocess.py의 3단계 Fallback 활용
     - CLI 실패 시 자동으로 REST API로 전환
     - 각 분대원이 독립 API 키 사용 → quota 독립

  C. 분대원별 API 키 환경변수 분리
     - GEMINI_API_KEY_A01=... GEMINI_API_KEY_A02=...
     - 각 분대원이 자신의 키 사용 (quota 완전 독립)
```

### Naver API

```
무료 한도: 25,000 calls/day
1,000명 × 150calls = 150,000 calls 필요

해결: Naver API 계정 6개 (각 25,000 = 총 150,000)
비용: Naver API 유료 플랜 없음 (계정 추가가 유일한 방법)
또는: 수집 데이터 캐싱 (같은 정치인 재수집 방지)
```

### Supabase 동시 연결

```
275명 동시 (시나리오 C):
  - 분대원 275개 × Supabase 연결 1개 = 275 연결
  - Supabase Pro: 200 연결 → ❌ 초과

해결:
  A. Supabase Pro + PgBouncer (연결 풀링) → 실제 연결 20개로 처리
  B. 분대원별 처리 시간 엇갈리게 (전부 동시 시작하지 않음)
  C. Supabase Enterprise (무제한 연결)
```

### Claude 평가 (분대원 = CLI Direct)

```
분대원 = Claude Code 에이전트 → 직접 평가 = CLI Direct
클로드 API 호출 없음 = Rate Limit 없음 (구독)

단, 주의사항:
  - 분대원이 /evaluate-politician-v40 스킬 사용
  - 평가 자체는 분대원 컨텍스트 내에서 처리
  - 배치 50개 기준, 10카테고리 = 2,000~3,000개 평가/명
```

## 구현 설계: V40 소대편제 워크플로우

### 소대장 임무 (Orchestrator)

```
소대장의 역할:
1. Supabase에서 처리 미완료 정치인 목록 조회
2. 소대별로 200명씩 분배
3. 각 분대장에게 SendMessage로 작업 배정
4. 진행 상황 모니터링 (Supabase processing_status 컬럼)
5. 실패 재처리 (failed → re-queue)
6. 최종 완료 보고

소대장이 직접 코딩 최소화:
- Python 스크립트는 이미 존재 (collect_*, validate_*, adjust_*, evaluate_* 등)
- 소대장은 분배 + 모니터링만 담당
```

### 분대장 임무 (Squad Lead)

```
분대장의 역할:
1. 소대장으로부터 20-40명 배정받음
2. 분대원 11명에게 각 1-4명씩 배분
3. 분대원 진행 상황 추적
4. 실패 시 재시도 or 소대장 보고
5. 완료 시 소대장에게 보고

분대원 투입 시 모델: haiku (단순 반복 작업)
```

### 분대원 임무 (Solo Politician Pipeline)

```python
# 분대원 1명의 실행 스크립트 (의사코드)

POLITICIAN_ID = "c45565d7"

# Phase 3: 평가 (4개 AI 병렬)
```

## 실행 계획 (단계별)

### Step 1: 인프라 준비

```
1. Supabase: processing_status 컬럼 추가
   ALTER TABLE politicians ADD COLUMN processing_status TEXT DEFAULT 'pending';

2. Gemini API 키 5개 확보 (각 분대별 1개)
   - 이미 보유한 키 + 추가 발급

3. Naver API 계정 6개 확보

4. Supabase Pro 플랜 전환 (연결 풀링)
```

### Step 2: 분대원용 실행 스크립트 제작

```
파일: V40/scripts/workflow/run_single_politician_v40.sh
입력: POLITICIAN_ID, POLITICIAN_NAME, GEMINI_KEY_IDX
동작: Phase 0~5 완전 자동 실행
출력: 성공/실패 JSON
```

### Step 3: 소대 편성 (C형 - 6소대)

```
소대 6개 × 분대 5개 × 분대원 11명 = 330개 동시 처리
소대 2라운드 → 660명
소대 3라운드 → 990명 + 잔여 10명

총 소요 시간: 3 × 60분 = 180분 = 3시간
```

### Step 4: 모니터링

```python
# 소대장이 주기적으로 진행 상황 확인
SELECT processing_status, COUNT(*)
FROM politicians
GROUP BY processing_status;

# 예상 출력:
# pending    | 670
# processing | 330
# completed  | 0
# → 1라운드 진행 중
```
