## 2. V40 Type C (n8n + API)의 구조적 병목

### 핵심 문제: SplitInBatches = 순차 처리

현재 `v40_type_c_workflow.json`의 실제 코드:

```json
{
  "name": "Collection Loop",
  "type": "n8n-nodes-base.splitInBatches",
  "parameters": { "batchSize": 1 }  ← 정치인 1명씩 처리!
}
```

**n8n의 SplitInBatches는 루프다. 병렬이 아니다.**

아무리 n8n을 써도 이 구조라면:

```
정치인 1번 완전 처리 → 정치인 2번 완전 처리 → ... → 정치인 N번
```

### Phase별 실제 소요 시간 분석 (정치인 1명 기준)

| Phase | Type A (현재 수동) | Type C (n8n API) | 병목 원인 |
|-------|-------------------|------------------|-----------|
| Phase 0 등록 | 15분 (수동) | 1분 (자동화 가능) | - |
| Phase 1 수집 | 40-55분 | **8-15분** ✅ | Gemini free: 15 req/min |
| Phase 2 검증 | 15분 | 3분 | - |
| Phase 2-2 조정 | 5-180분 | 5-30분 | 재수집 라운드 수 |
| Phase 3 평가 | **4-8시간** ❌ | **25-40분** ✅ | 4 AI 병렬 HTTP Request |
| Phase 4-5 점수/보고서 | 30분 | 5분 | - |
| **합계** | **5-10시간** | **47-94분** | |

**Type C가 Type A 대비 6-8배 빠른 이유**: Phase 3에서 4개 AI를 HTTP Request로 병렬 호출

### 그런데도 1000명이 문제인 이유

```
Type C 시간 × 정치인 수 = 총 소요 시간

1명: 약 70분
1000명 순차: 70분 × 1000 = 70,000분 = 1,167시간 = 49일
```

n8n을 써도 49일. 의미가 없다.

## 3. Phase별 세부 병목 분석

### Phase 1: 수집 - Gemini API Rate Limit

```
Gemini 2.0 Flash (무료): 15 req/min
목표: 정치인 1명 × 10카테고리 × 6라운드 = 120회 API 호출

120 / 15 = 8분 (무료 tier, 1명)
1000명 × 8분 = 8,000분 = 133시간 (순차)

Gemini paid tier (1000 req/min):
120 / 1000 = 0.12분 = 7초 (1명)
1000명 × 7초 = 7,000초 = 117분 (순차)
```

→ **유료 전환 시 수집 병목 거의 해소** (117분으로 단축)

### Phase 3: 평가 - n8n 내부 구조 문제

Type C 워크플로우에서 평가 루프 구조:

```
정치인 Loop (batchSize=1, 순차)
  └─ 카테고리 Loop (순차)
       └─ 배치 Loop (25개씩, 순차)
            ├─ Claude API 호출 ──┐
            ├─ ChatGPT 호출     ├─ 이 4개는 병렬
            ├─ Gemini API 호출  │
            └─ Grok API 호출   ──┘
```

**병렬인 것**: 1명의 1카테고리에서 4개 AI 호출 (4배 속도 향상)
**순차인 것**: 정치인 간, 카테고리 간, 배치 간 (전부 순차)

### Phase 3 실제 계산

```
1명 × 10카테고리 × 4배치(각 25개) × 4 AI = 160 API 호출
(단, 4 AI는 병렬이므로 실제 순차 호출 수 = 40 라운드)

라운드당 응답시간: 약 5-10초
40라운드 × 7.5초 = 300초 = 5분 (이상적)

실제: n8n 노드 오버헤드, DB 저장 등 포함 → 약 20-30분/명

1000명 × 25분 = 25,000분 = 417시간 = 17일 (순차)
```

### Phase 5: 보고서 생성 - 미자동화

현재 Type C 워크플로우에 보고서 생성(HTML)이 **구현되지 않음**.
HTML 생성은 여전히 수동 + copywriter 리뷰 필요.

## 4. n8n 도입의 실제 가치

### 가치 있는 부분 ✅

| 항목 | 효과 |
|------|------|
| Phase 3 AI 병렬 호출 | 4개 AI 동시 실행 → 4배 향상 |
| Phase 1-2 자동화 | 수동 개입 제거 |
| 오류 자동 복구 | Gate 노드로 실패 시 재시도 |
| 상태 관리 | Phase 진행 상태 DB 저장 |
| 스케줄링 | 야간/주말 자동 실행 |

### 가치 없는 부분 ❌ (1000명 관점에서)

| 항목 | 이유 |
|------|------|
| 정치인 간 병렬화 | n8n SplitInBatches는 순차 루프 |
| 1000명 동시 처리 | 단일 n8n 인스턴스 불가 |
| 보고서 HTML 생성 | Type C에 미구현 |

**결론**: n8n은 좋은 도구지만, 현재 설계로는 "자동화된 순차 처리"일 뿐.
1000명을 "동시에" 처리하려면 근본적인 아키텍처 변경이 필요.

## 5. 1000명 동시 처리를 위한 실제 병목과 해결책

### 병목 1: 순차 처리 구조

**문제**: n8n 단일 인스턴스 SplitInBatches
```
현재: [P1] → [P2] → [P3] → ... → [P1000]  ← 49일
```

**해결**: n8n Sub-workflow 병렬 실행
```
n8n Execute Workflow 노드 (병렬 옵션 활성화):
[P1~P10] 동시 → [P11~P20] 동시 → ... → 10배 속도

결과: 49일 → 5일
```

또는 **n8n Queue 모드** (Enterprise):
```
Worker 10개 동시 실행:
49일 → 5일
```

### 병목 2: Gemini API Rate Limit (수집)

**무료 tier**: 15 req/min
**해결책**:
1. Gemini API 유료 tier 전환 (1000 req/min) → 133시간 → 2시간
2. 또는 API 키 여러 개 로테이션 (10개 키 = 150 req/min)

### 병목 3: Claude = CLI Direct ($0)

현재 Claude 평가는 CLI Direct (구독 무제한).
Type C에서는 API 사용 → 비용 발생.

**1000명 API 비용 추산**:
```
Claude Haiku 4.5:
  10카테고리 × 4배치 × 3K 입력토큰 = 120K 입력/명
  120K × 1000명 = 120M 토큰
  $0.80/MTok → $96

ChatGPT o4-mini:
  비슷한 규모 → $150-200

Gemini Flash:
  수집 + 평가 합산 → $50

Grok 3:
  $3/MTok 기준 → $360

총 API 비용: 약 $600-700 / 1000명
```

비용 자체는 감당 가능하나, **Claude는 CLI Direct 유지**가 최적:
```
Claude CLI Direct: $0 (구독)
Claude API: $96
→ CLI 유지 권장
```

→ Claude만 Type B 방식 (CLI + Wait 노드) 사용하면 비용 $500 이하로 절감

### 병목 4: DB 쓰기 속도 (Supabase)

1000명 × 1200개 수집 데이터 = 120만 행 삽입
1000명 × 4800개 평가 결과 = 480만 행 삽입

**Supabase Free Tier**: 500MB DB → 부족할 수 있음
**해결**: Supabase Pro Tier 또는 벌크 삽입 최적화 필요

## 6. 1000명 처리를 위한 권장 아키텍처

```
[정치인 목록 DB]
       ↓
[n8n Scheduler: 매 10분마다]
       ↓
[미처리 정치인 10명씩 배치 추출]
       ↓
[n8n: 10개 Sub-workflow 병렬 실행]
  ├─ [Sub-workflow 1: 정치인 A]
  │    Phase1→2→2-2→3→4
  ├─ [Sub-workflow 2: 정치인 B]
  ...
  └─ [Sub-workflow 10: 정치인 J]
       ↓
[완료 표시 후 다음 배치]
       ↓
[보고서 생성: 완료된 정치인만 순차]
```

**예상 처리 속도**:
- 10명 병렬, 1명당 70분 → 10명 = 70분
- 1000명 / 10 = 100 배치 × 70분 = 7000분 = 117시간 = **5일**

**추가 최적화** (30명 병렬, Gemini 유료):
- 1000명 / 30 = 34 배치 × 20분 = 680분 = **약 11시간**

## 7. 즉시 적용 가능한 개선사항

### 단기 (현재 Type C 개선)

1. **Collection Loop `batchSize` 유지**: 이미 올바름 (1명씩 API 제한 준수)
2. **Sub-workflow 병렬화 추가**: Execute Workflow 노드로 10명씩 병렬

```json
// n8n Execute Workflow 노드 추가
{
  "name": "Parallel Sub-workflows",
  "type": "n8n-nodes-base.executeWorkflow",
  "parameters": {
    "source": "database",
    "workflowId": "V40_Single_Politician",
    "mode": "each",  ← 각 아이템을 별도 실행으로
    "waitForSubWorkflow": false  ← 비동기 병렬
  }
}
```

3. **Phase 5 HTML 보고서 생성 자동화**: 현재 미구현, Code 노드로 추가 필요

### 중기 (안정화 후)

1. Gemini API 유료 전환 (수집 속도 66배 향상)
2. Claude만 CLI Direct 유지 (비용 $0 유지)
3. Supabase Pro 전환 (DB 용량)

### 장기 (1000명 운영)

1. n8n Enterprise Queue Mode 또는 Worker 분산
2. 전용 서버에 n8n 자체 호스팅 (메모리 확장)
3. 처리 상태 대시보드 구축
