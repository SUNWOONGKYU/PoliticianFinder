# 🏗️ Grid vs Voxel: 아키텍처 정확한 분류

**작성일**: 2025-10-21
**중요도**: 🔴🔴🔴 기술 정의의 정확성
**주제**: 우리 시스템이 "그리드"인지 "복셀(Voxel)"인지 기술적 분석

---

## 📚 용어 정의

### 1️⃣ Voxel (복셀)의 개념

```
Voxel = Volume + Pixel
       = 3차원 공간의 최소 단위 큐브

특징:
✓ 3D 공간의 각 격자 셀에 "완전한 데이터"를 저장
✓ 색상, 밀도, 투명도 등 "모든 정보"를 각 셀에 포함
✓ 셀 내에 필요한 "모든 속성"을 저장

메모리 특징:
- 각 셀: 많은 데이터 포함
- 저장 크기: 큼 (50-500 바이트/셀)
- 활용도: 높음 (모든 정보 한곳에)

사용 분야:
- 의료 영상 (CT, MRI) - 각 복셀에 밀도, 색상 등 저장
- 3D 게임 엔진 - 각 복셀에 텍스처, 재질 등 저장
- 과학 시뮬레이션 - 각 복셀에 온도, 압력 등 저장
```

### 2️⃣ Grid (그리드)의 개념

```
Grid = 3차원 좌표계 기반 색인 체계

특징:
✓ 각 셀은 "최소한의 인덱스 정보"만 저장
✓ 셀의 위치: (X, Y, Z) 좌표
✓ 셀의 내용: 외부 데이터 파일 참조 (포인터/링크)
✓ 실제 데이터: 별도 저장소에 저장

메모리 특징:
- 각 셀: 최소한의 정보만 (포인터, ID, 상태)
- 저장 크기: 작음 (10-50 바이트/셀)
- 활용도: 높음 (필요한 것만 참조)

사용 분야:
- 데이터베이스 색인 시스템
- 맵 타일 시스템 (게임의 지도)
- 공간 분할 시스템 (R-tree, Quad-tree)
```

### 저장소 구조 비교

```
항목              Voxel           우리 그리드      결론
───────────────────────────────────────────────────
셀 내용          완전한 데이터    인덱스+포인터    ✅ 그리드
```

## 🎯 우리 시스템이 그리드인 이유

### 5가지 기술적 증거

```
증거 1: 최소한의 인덱싱 정보만 저장
────────────────────────────
셀(T, S, W)에 저장되는 것:
- Task ID (문자열)
- 상태 (enum)
- 담당자 (ID)
- 진도율 (%)
- 파일 참조 (경로)

→ 전형적인 "그리드 색인" 구조
→ 복셀이면 이 모든 것 + 상세 내용까지 저장해야 함
```

```
증거 2: 모든 상세 정보가 외부 파일에 저장
────────────────────────────────────────
그리드 셀: 포인터만
실제 데이터: 별도 파일
├─ P1F1.md (작업 설명)
├─ P1F1.py (코드)
├─ P1F1_test.md (테스트)
├─ P1F1_blocker.md (블로커 분석)
└─ P1F1_design.md (설계)

→ 전형적인 "파일 시스템 기반 그리드"
→ 각 셀이 파일 시스템을 참조하는 구조
```

```
증거 3: 모듈식 참조 구조
────────────────────
CSV 셀 (P1F1):
{
  "id": "P1F1",
  "ref_list": [
    "tasks/P1F1.md",
    "code/P1F1.py",
    "tests/P1F1_test.md"
  ]
}

→ 각 셀이 "링크" 또는 "포인터"를 가짐
→ 실제 데이터는 분산 저장
→ 전형적인 "그리드 + 파일 시스템 통합"
```

```
증거 4: 메모리 효율성
────────────────────
그리드 구조:
- CSV 셀: 1KB (인덱스만)
- 외부 파일: 필요할 때만 로드

복셀 구조였다면:
- 각 셀: 10-50KB (모든 데이터)
- 512×512×512 = 135M 셀
- 전체: 1.35TB-6.75TB

우리 구조:
- CSV: 512×512×512 = 135M 셀 × 1KB = 135GB (이론)
- 실제: Phase 1-5만 = 약 100-200MB (매우 작음)
- 파일들: 필요할 때만 접근

→ 메모리 효율: 100배 이상 우월
→ 확장성: 매우 우수
```

```
증거 5: 확장성과 유연성
────────────────────
그리드 방식:
- 새 파일 추가: 쉬움 (포인터만 추가)
- 셀 크기: 작음 (고정)
- 변경 영향도: 낮음 (각 셀 독립)

복셀 방식이었다면:
- 새 정보 추가: 어려움 (셀 전체 재구성)
- 셀 크기: 변경됨 (가변)
- 변경 영향도: 높음 (전체 재계산)

→ 우리 방식이 훨씬 확장 가능
```

## 💡 PublicityLabs와의 논의

### 기술 용어의 정확성

```
PublicityLabs의 질문:
"이게 Grid냐 Voxel이냐?"

우리의 답변 (기술적 근거):

❌ Voxel이 아닌 이유:
1. 셀에 완전한 데이터를 저장하지 않음
2. 각 셀에 모든 속성을 포함하지 않음
3. 메모리 모델이 다름
4. 확장 방식이 다름

✅ Grid인 이유:
1. 3D 좌표 기반 색인 시스템
2. 각 셀은 최소 정보만 저장
3. 실제 데이터는 외부 저장소 참조
4. 파일 시스템과 통합된 포인터 구조
5. 매우 효율적인 메모리 활용

용어 정확성:
우리 시스템 = "3D Grid"가 맞다
            (Indexed File System Grid)
```

```
만약 "복셀"이라 표현했다면:
```

## 🎯 결론: 기술 정의의 정확성

### 우리 시스템의 정확한 분류

```
❌ 틀린 표현:
"3D Voxel Project Management System"

✅ 정확한 표현 (선택 가능):
1. "3D Grid Project Management System"
2. "Indexed 3D Grid File System"
3. "3D Coordinate-based Index Grid"
4. "Sparse 3D Grid (희소 그리드)"

가장 정확한 표현:
"3차원 좌표 기반 그리드 색인 시스템으로
파일 시스템을 관리하는 프로젝트 관리 방식"
```

### 특허 청구항에서의 활용

```
기존 표현:
"3차원 그리드 구조를 이용한 프로젝트 관리 방법"

강화된 표현:
"3차원 좌표계 (시간축, 공간축, 작업축) 기반의
그리드 색인 시스템으로 프로젝트 정보를 관리하되,
각 그리드 셀에는 최소한의 인덱스 정보만 저장하고
실제 작업 정보는 별도 파일에 저장하여
메모리 효율성과 확장성을 극대화하는
프로젝트 관리 방법"

효과:
↑ 기술 명확성 향상
↑ 특허 범위 확대
↑ 심사관의 이해도 증진
↑ 특허 승인 확률 향상
```
